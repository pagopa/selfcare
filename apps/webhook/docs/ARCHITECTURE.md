# Webhook Service - Architecture

## Required Architecture Inputs

- `Requirements source: REQUIREMENTS.md`
- `System purpose: configure HTTP webhook endpoints for Selfcare products, deliver asynchronous notifications to active webhooks, persist delivery state, and retry undelivered notifications.`
- `Primary use cases: create webhook; list webhooks; get webhook by product; update webhook by product; request notification delivery for a product; process notification retries; expose application version, health, OpenAPI, and Swagger UI.`
- `Target users / actors: external clients managing webhook configurations; internal Selfcare services requesting notification delivery; external target endpoints receiving webhook calls; application scheduler processing retries.`
- `Runtime environment: Azure Container Apps in the ar domain for dev, uat, and prod; container listens on port 8080 with Quarkus health probes on /q/health/live, /q/health/ready, and /q/health/started.`
- `Server framework: Java 17 Quarkus service using REST APIs, SmallRye Mutiny reactive flows, Quarkus Scheduler, Quarkus MongoDB Panache, SmallRye JWT, and SmallRye OpenAPI.`
- `Client framework: no UI client framework; outbound HTTP delivery uses a server-side Vert.x WebClient.`
- `API style and integration model: JSON REST APIs published through Azure API Management at base path external/webhook using the generated OpenAPI document; internal notification API triggers outbound HTTP webhook calls to configured target endpoints.`
- `Authentication and session model: JWT bearer authentication support through Quarkus SmallRye JWT and selfcare-sdk-security; JWT_PUBLIC_KEY is provided as a Key Vault-backed Container App secret; subject is read from the uid claim; no stateful server session is defined.`
- `Data model expectations: Cosmos DB Mongo API database selcWebhook with webhooks and webhookNotifications collections; unique index on webhooks._id and webhooks.productId; non-unique index on webhooks.products and webhookNotifications.webhookId; webhookNotifications has a 30-day TTL.`
- `Deployment model: Terraform-managed Azure resources under /infra/resources/webhook; Container App image ghcr.io/pagopa/selfcare-webhook-ms:<image_tag>; secrets sourced from Azure Key Vault; API exposed through Azure API Management and routed to the private Container App DNS name.`
- `Scale expectations: Container App cron scale rule for business hours, default min/max replicas 1/1, dev overrides min replicas to 0, prod overrides max replicas to 5, desired replicas to 3, CPU to 1.25, and memory to 2.5Gi. Retry scheduler processes at most 100 notifications per cycle with a 5-minute MongoDB processing lock.`
- `Security expectations: JWT public key and MongoDB connection string are secrets; webhook header encryption key and IV are secrets; webhook headers are encrypted at rest; insecure ingress is disabled; application exposes HTTPS-compatible ingress through Azure Container Apps/APIM; relevant string inputs must be sanitized before use or logging.`

## Initial Architecture (Provisional)

The service is a single Quarkus microservice deployed as an Azure Container App behind Azure API Management. It exposes JSON REST endpoints for webhook configuration management, notification submission, and application version reporting. OpenAPI and Swagger UI are generated by the service and APIM imports the generated OpenAPI document.

Webhook configuration requests are handled by a REST controller and application service boundary. Webhook data is persisted in Cosmos DB Mongo API through reactive MongoDB Panache repositories. Webhook headers are encrypted before persistence and decrypted when included in API responses.

Notification submission is asynchronous from the caller perspective: the API accepts a product-level notification request, finds active webhooks for that product, creates notification records, and starts HTTP delivery through Vert.x WebClient. Delivery success is represented by a target response with status code `2xx`; non-`2xx` responses and communication errors update notification state for retry or permanent failure.

Retry processing runs inside the same service process through Quarkus Scheduler. The scheduler periodically claims pending or retryable notification records with MongoDB-side locking so multiple replicas do not process the same notification concurrently. Each scheduler cycle is bounded to 100 records and lock duration is 5 minutes.

Assumptions:

- The service remains a single deployable microservice unless requirements introduce separate worker or queue components.
- APIM is the external API entry point for documented APIs; direct Container App ingress exists but routes through the private Container App DNS name configured in APIM.
- Cosmos DB Mongo API is the production persistence target, while the application configuration remains compatible with generic MongoDB connection strings.
- Retry delay/backoff fields are part of the data model but are not architecture-driving until retry timing is implemented.

Open decisions:

- Authorization policy by endpoint is TO BE DECIDED; requirements define JWT support but not roles, scopes, or APIM policies.
- Webhook target allowlist/denylist, egress restrictions, and SSRF protection are TO BE DECIDED.
- Whether `POST /webhooks/notify` must return only after initial delivery is attempted or only after records are persisted is TO BE DECIDED.
- Delete webhook behavior and retention semantics are TO BE DECIDED because the API currently returns `501 Not Implemented`.
- Retry backoff scheduling semantics are TO BE DECIDED because stored delay fields are not currently used.

## Dependency Rules

- Do not add a dependency when the standard library or a few lines of first-party code
will do.
- Prefer zero new dependencies. If a library is required, justify it in the PR
description.
- Only use libraries that are actively maintained (commit or release within the last 12
months).
- Only use the latest stable major version. No deprecated, abandoned, or pre-release
packages.
- Reject any library with known unpatched CVEs. Check before adding and on every update.
- Audit transitive dependencies, not just direct ones. A small direct dep with a large or
unvetted tree is a rejection.
- Pin exact versions with a committed lockfile. No floating ranges in production.
- Prefer libraries with a narrow scope, minimal dependencies of their own, and a clear
security track record.
